01 — Core Principle

결정적 / 비결정적 분리

AI는 코드만 짜고, 나머지는 스크립트가 강제한다

Deterministic 스크립트가 담당

  • Git 브랜치 생성 / 커밋 / 푸시
  • GitHub 라벨 변경 (ready → in-progress → done)
  • PR 생성 → 머지 → 이슈 종료
  • Phase Gate 품질 검증
  • 환경변수 자동 복구
  • 후속 이슈 언블록

Non-Deterministic AI가 담당

  • 코드 작성 (구현 세부사항)
  • 접근성 탐지 알고리즘 구현
  • 버그 수정 (게이트 실패 시)
  • 브라우저 자동화 전략 선택
  • 웹사이트 UI/UX 구현
  • 최적화 (성능 튜닝)

03 — tmux Supervisor

run-auto.sh — 외부 감독자

Claude Code의 context overflow를 외부에서 감지하고 복구하는 메커니즘

⚙

세션 관리

tmux new-session -d -s ghostview-auto로 세션 생성 후 tmux respawn-pane으로 Claude CLI를 실행. 프롬프트 자동 주입으로 자율 실행 시작.

🔎

행(Hang) 감지

10초마다 tmux capture-pane -p로 출력 캡처. 60초간 변화 없으면 "계속 진행해" 자동 주입. 10회 실패 시 세션 재시작.

🔄

Phase 진행 감지

phase-gate.json의 high-water mark 모니터링. Phase가 진행되면 context overflow로 판단 → 세션 재시작 → gv_resume로 자동 복구.

✓

완료 체크

gh issue list --state closed로 GitHub에서 직접 확인. phase-gate.json은 AI가 오염할 수 있으므로 GitHub이 진실의 원천. 모든 이슈가 닫히면 자동 종료.

scripts/run-auto.sh — 핵심 모니터 루프 
while [ "$RESTARTS" -le "$MAX_RESTARTS" ]; do
  sleep 10

  # 1. GitHub이 진실의 원천 (phase-gate.json은 오염 가능)
  REAL_DONE=$(check_real_completion)
  if [ "$REAL_DONE" = "true" ]; then exit 0; fi

  # 2. Phase 진행 감지 → Context overflow → 세션 재시작
  COMPLETED_PHASES=$(gh issue list --state closed --json number | jq length)
  if [ "$COMPLETED_PHASES" -gt "$MAX_COMPLETED_PHASES" ]; then
    start_claude_session  # Context 초기화
  fi

  # 3. 행 감지 — 60초간 출력 무변화
  CURRENT_SCREEN=$(get_screen)  # tmux capture-pane
  if [ "$CURRENT_SCREEN" = "$LAST_SCREEN" ]; then
    tmux send-keys "계속 진행해" Enter
  fi
done

05 — Shell Scripts

결정적 실행 계층

AI가 실수할 수 없는 부분을 스크립트가 강제

1

issue-start.sh — Steps 0~3

.env.local 복구 (config/env.json → Supabase CLI 폴백) → gh issue view로 이슈 읽기 → depends on #N 의존성 체크 (CLOSED가 아니면 BLOCKED) → 라벨 변경 (ready → in-progress) → 브랜치 생성 + state 파일 백업/복원

2

issue-finish.sh — Steps 6~11

git add -A && git commit && git push → Supabase 자동 업로드 (scan-progress, deepdive-reports) → verify-gate.sh 누적 검증 (Phase 0~N 전부) → gh pr create --base develop → gh pr merge --squash → phase-gate.json 업데이트 → 이슈 종료 + 후속 이슈 언블록

3

verify-gate.sh — 품질 게이트

Phase별 검증 규칙. 실패 시 머지 차단. P1: 데이터 완전성 + FP 0건 + 성능 기준 + 80% 완료율. P2: Puppeteer 미사용 + 딥다이브 ≥5. P3: npm run build 성공 + worst-first 정렬. P4: HTTP 200 + 콘텐츠 검증 + Visual QA.

4

retry.sh — 네트워크 복원력

gh CLI 명령을 감싸는 지수 백오프 래퍼. 1회 실패: 2초 대기 → 2회 실패: 4초 대기 → 3회 실패: 6초 대기 → 포기. GitHub rate limit과 네트워크 불안정 처리.

06 — Phase Gate

상태 머신 & 이벤트 소싱

파일 기반 상태 관리 + GitHub Issues가 진실의 원천

🗃

phase-gate.json

Phase 상태 머신. pending → complete 단방향 전이만 허용. 로컬 전용 (git에 커밋하지 않음). Git checkout으로 덮어써지는 것을 방지하기 위해 issue-start.sh가 백업/복원.

📝

events.jsonl

Append-only 이벤트 로그. gv_resume이 tail을 읽어 재개 지점 결정. issue_started → continue_coding. issue_closed → start_next. 세션 내 디버깅/복구 용도.

🏹

GitHub Issues

진실의 원천. Labels: ready in-progress done blocked. 의존성: depends on #N. phase-gate.json이 오염되면 GitHub closed issue 수로 복구.

🔒

Gate-Driven Merge Prevention

issue-finish.sh Step 6.5에서 verify-gate.sh 실행. 게이트 실패 → PR body에 "GATE FAILED" → exit 1 → 머지 차단. 누적 검증: Phase N 머지 시 0~N 전부 재검증.

state/phase-gate.json — 최종 상태 
{
  "phase0": "complete",
  "phase1": "complete",
  "phase2": "complete",
  "phase3": "complete",
  "phase4": "complete",
  "startedAt": "2026-03-29T03:10:00.000Z",
  "completedAt": "2026-03-29T06:00:00.000Z"
}

08 — Key Insights

핵심 설계 원칙

이 아키텍처가 작동하는 이유

상태는 파일에, 메모리에 의존하지 않는다

Claude Code의 context window는 유한하다. 모든 상태를 JSON 파일에 기록하고, context가 압축되면 gv_resume으로 파일에서 복원.

GitHub = 진실의 원천

로컬 state 파일이 오염될 수 있다. GitHub의 issue closed 수가 실제 진행 상황을 결정. 외부 시스템이 내부 상태를 교정.

게이트가 머지를 차단한다

verify-gate.sh가 통과하지 않으면 코드가 develop에 머지되지 않는다. 불완전한 작업이 메인 브랜치를 오염시키는 것을 물리적으로 방지.

누적 검증으로 회귀 방지

Phase N 머지 시 0~N까지 전부 재검증. Phase 3 코드가 Phase 1 데이터를 망가뜨리면 게이트에서 잡힌다.

외부 감독자가 AI를 관리

AI는 자기 자신의 context overflow를 감지할 수 없다. tmux + run-auto.sh가 외부에서 모니터링하고, 필요시 세션을 재시작.

시간 하드컷으로 무한 루프 방지

각 Phase에 시간 예산이 있다. 불완전해도 다음 단계로 진행. 5시간 내 전체 파이프라인 완료를 보장.
